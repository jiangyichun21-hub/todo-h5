# Vibe Coding 体验总结 — 用 AI 从零构建多用户待办应用

## 1. 项目概述

用自然语言描述需求，由 AI（Claude Code）逐步实现一个完整的多用户待办事项 Web 应用。从空白项目到功能上线（本地运行），全程以对话方式推进，不手动编写代码。

**核心功能：**
- 昵称 + 密码注册 / 登录
- 待办事项的增删改查
- 完成状态切换（带动画）
- 智能排序（未完成优先，组内按时间倒序）
- 底部筛选栏（全部 / 进行中 / 已完成）
- 多用户数据隔离（基于 Supabase Auth）
- 移动端优先的响应式设计
- 中文界面

## 2. 技术栈

| 层 | 技术 |
|---|---|
| 框架 | Next.js 16（App Router） |
| 语言 | TypeScript 5 |
| 样式 | Tailwind CSS 4 |
| 后端 + 数据库 | Supabase（PostgreSQL + Auth） |
| 状态管理 | React `useState`（纯客户端） |
| 部署 | 尝试 Vercel（npm install 持续失败） |

**选择理由：**
- Next.js + Tailwind 是当前最流行的全栈组合，零配置即可用
- Supabase 提供免费托管的 Postgres 数据库和开箱即用的 Auth，不用自己搭后端
- 纯客户端状态管理保持简单，适合这种小型应用

## 3. 开发过程回顾

### 3.1 环境搭建

项目目录已有一个 `create-next-app` 初始化的骨架，直接在上面开发。依赖安装正常，`npm run dev` 启动正常。

**耗时：** 几乎零成本。

### 3.2 页面骨架

第一轮对话就完成了基础功能：
1. 输入框 + 添加按钮
2. 待办事项列表渲染
3. 每项的删除和完成切换
4. 底部筛选栏

**感受：** AI 一次性写出完整的组件代码，功能点全覆盖，不需要反复修改。

### 3.3 移动端适配

从桌面布局改为移动端优先：
- 触摸友好的按钮尺寸（`py-3`）
- 底部固定筛选栏（类似原生 App Tab Bar）
- 卡片式列表项，圆角 + 阴影
- `active` 态视觉反馈

**迭代次数：** 2-3 轮，主要是布局细节调整。

### 3.4 核心功能增强

**排序规则：**
- 未完成在前，已完成在后
- 同组内按关联时间倒序

**时间戳展示：**
- 每条待办显示创建时间
- 完成状态额外显示完成时间

**状态管理：**
- 从 `useEffect` + `localStorage` 改为纯 `useState` 惰性初始化
- 封装 `saveTodos` 函数，每次操作同步更新 state 和 localStorage

### 3.5 数据库接入和用户认证

这是最大的一次架构变更：

1. **接入 Supabase SDK** — `npm install @supabase/supabase-js`
2. **创建 `lib/supabase.ts`** — 客户端封装
3. **创建登录页面** — `/login` 路由
4. **改写主页** — 用 Supabase API 替代 localStorage

**认证方案：**
- 用户输入昵称 + 密码
- 后端自动生成随机邮箱（`user_xxx@todo.app`）注册到 Supabase Auth
- 昵称存储在 `profiles` 表中，通过映射实现昵称登录
- 关闭邮箱确认，注册后直接登录

**踩坑：**
- Supabase 的 `Email` 认证频率限制导致 `email rate limit exceeded`，需要关闭邮箱确认
- 注册时 `profiles` 表写入失败 → 改用 `upsert` 操作
- RLS 策略需要手动配置，否则无法读取 profiles 数据

### 3.6 部署尝试

Vercel 部署遇到 `npm install` 持续失败（`Exit handler never called!`），多次尝试未解决。

**可能原因：** Vercel 服务器与 npm registry 的网络连接问题，与代码本身无关。

## 4. Claude Code 使用体验

### 4.1 AI 理解意图的准确度

**整体评价：** 约 80% 的情况下能准确理解意图并一次性写对。

**表现好的场景：**
- 描述具体功能时（"每个事项有删除按钮"）
- 描述交互行为时（"点击切换完成/未完成"）
- 描述布局要求时（"居中布局，圆角卡片"）
- 语言偏好（"页面用中文"）被完整记住

**理解偏差的场景：**
- 模糊描述颜色时（"淡紫色、低饱和、莫兰迪"）需要多轮微调
- "注册成功后自动登录" 最初理解成提示语而非自动跳转
- 对 Supabase 限流错误的排查，AI 需要人工提供日志才能定位

### 4.2 对话轮次统计

| 功能 | 轮次 | 说明 |
|---|---|---|
| 基础 Todo 功能 | ~2 轮 | 一次性写全四个需求点 |
| 移动端适配 | ~3 轮 | 布局 + 触摸反馈 + 空状态 |
| 中文界面 | ~1 轮 | 替换所有文本 |
| 排序 + 时间戳 | ~2 轮 | 一次实现，一次小修 |
| 状态管理优化 | ~1 轮 | 去掉 useEffect |
| Supabase 接入 | ~8 轮 | 包含注册/认证/RLS/限流排查 |
| UI 美化 | ~3 轮 | 配色调整（橙色→淡紫） |
| 部署 | ~5 轮 | 持续失败 |

**平均每个功能约 2-3 轮对话完成。** 复杂功能（数据库接入）需要更多轮次。

### 4.3 代码质量评价

**优点：**
- 组件结构清晰，逻辑分离
- TypeScript 类型定义完整
- 错误处理到位（中文错误映射）
- Tailwind 类名组织合理
- 没有冗余代码或死逻辑

**可以改进的：**
- 部分组件较长（~200 行单文件），可以拆分
- 登录逻辑耦合在同一个组件中
- 缺少测试
- 环境变量缺少时没有 graceful fallback

### 4.4 遇到的问题和解决方式

| 问题 | 原因 | 解决方式 |
|---|---|---|
| API 限流错误 | 对话太频繁触发 429 | 等待或提高配额 |
| Email rate limit | Supabase 发邮件频率限制 | 关闭 Confirm email |
| 昵称登录失败 | profiles 表没写入 | 改用 upsert + 检查 RLS |
| Vercel 部署失败 | npm install 网络超时 | 未解决 |

## 5. 总结

### 5.1 AI 擅长什么

- **页面搭建和 UI 实现** — 描述清楚样式需求，一次性产出可用组件
- **功能实现** — 增删改查、筛选、排序等 CRUD 逻辑，描述即代码
- **技术选型建议** — 能根据场景推荐合适的方案并说明取舍
- **错误排查** — 给出排查方向和解决步骤（前提是有足够日志）
- **快速迭代** — 改颜色、调布局、替换文本，秒级完成

### 5.2 AI 不擅长什么

- **环境 / 网络问题** — Vercel 部署失败无法远程修复，只能给建议
- **需要外部系统配合** — Supabase 配置、RLS 策略需要人工在后台操作
- **模糊审美描述** — "莫兰迪低饱和" 这种主观描述需要多次迭代
- **跨文件架构设计** — 倾向于写单文件大组件，不会主动拆分

### 5.3 感受：爽在哪、卡在哪

**爽点：**
- 从想法到可用界面不到 30 分钟
- 不需要手写一行代码，全程自然语言描述
- UI 细节调整即时生效，反馈循环极短
- 多语言支持、类型安全等"非核心需求"也能顺便搞定

**卡点：**
- API 限流是最烦的，连续问问题会触发，被迫等待
- Supabase 后台配置（RLS、邮箱确认）必须人工操作，AI 帮不了
- Vercel 部署失败卡了很久，问题不在代码而在基础设施
- 多次重试时容易重复触发限流

### 5.4 对 Vibe Coding 的整体评价

**结论：Vibe Coding 对于原型开发和小项目来说已经足够好用。**

用一个下午的时间，从零搭建了一个完整的多用户 Web 应用，包含数据库、认证、CRUD、移动端适配。如果排除部署环节，开发体验远超预期。

**适合的场景：** 原型验证、个人项目、小型 SaaS、学习项目、Hackathon
**不太适合的场景：** 企业级应用、需要复杂架构的项目、有严格合规要求的项目

**一句话总结：** 写代码本身已经不是门槛了，真正的瓶颈在环境配置、外部服务对接和部署上线。
